# A2팀 생성형 AI 통합 엔진 PM 사전 정리

작성일: 2026-06-26  
작성자: 손찬용 PM  
문서 목적: 6월 30일 팀 작업 시작 전, PM 기준에서 확정된 방향과 2주차 Pass 범위를 정리한다.

---

## 1. 현재 확정 사항

### 1.1 이미지 생성 API 기준

이미지 생성 API는 **GPT Image 2.0** 기준으로 진행한다.

| 항목 | 결정 내용 |
| --- | --- |
| 이미지 생성 모델 | GPT Image 2.0 |
| API 기준 모델명 | `gpt-image-2` |
| 2주차 중간점검 핵심 기능 | 프롬프트 기반 이미지 생성 |
| 우선 구현 범위 | 프롬프트 입력, 이미지 생성 API 호출, 결과 표시, 이미지 저장, 라이브러리 조회 |

### 1.2 기존 문서 간 차이 정리

초기 프로젝트 명세서에는 Gemini / Nano Banana 계열 모델이 함께 언급되어 있었으나, A2팀의 현재 PRD와 비용 산정 문서 기준으로는 **GPT Image 2.0을 이미지 생성 메인 API로 사용**하는 방향으로 정리한다.

따라서 이후 개발 문서, API 명세, 역할별 작업표에서는 이미지 생성 API를 `GPT Image 2.0 / gpt-image-2` 기준으로 통일한다.

---

## 2. 2주차 중간점검 Pass 범위

2주차 중간점검은 Pass / Fail 기준이므로, 전체 기능을 많이 보여주는 것보다 **프롬프트 → 이미지 → 저장 → 라이브러리 조회 흐름이 실제로 동작하는지**가 가장 중요하다.

### 2.1 필수 구현 범위

| ID | 기능 | 설명 | 완료 기준 |
| --- | --- | --- | --- |
| M001 | 이미지 프롬프트 입력 | 사용자가 이미지 생성용 프롬프트를 입력한다. | 텍스트 입력 가능 |
| M002 | 이미지 생성 API 호출 | `gpt-image-2` 기반 이미지 생성 API를 호출한다. | 테스트 프롬프트로 이미지 생성 성공 |
| M003 | 이미지 결과 표시 | 생성된 이미지를 화면에 표시한다. | 생성 결과가 결과 영역에 표시 |
| M004 | 이미지 저장 | 생성 이미지를 라이브러리에 저장한다. | 저장 버튼 클릭 후 저장 상태 반영 |
| M005 | 이미지 라이브러리 조회 | 저장된 이미지를 다시 조회한다. | 라이브러리 화면에서 저장 이미지 확인 |
| M006 | 기본 화면 설계 | 생성 화면, 결과 화면, 라이브러리 화면을 구성한다. | 생성 흐름을 이해할 수 있는 UI 완성 |

### 2.2 Pass 판단 기준

아래 흐름이 끊기지 않고 동작하면 2주차 중간점검 Pass 기준을 충족한 것으로 본다.

```text
프롬프트 입력
→ 이미지 생성 버튼 클릭
→ 이미지 생성 API 호출
→ 생성 결과 화면 표시
→ 저장 버튼 클릭
→ 라이브러리에서 저장 이미지 조회
```

### 2.3 2주차에서 후순위로 둘 기능

아래 기능은 중요하지만 2주차 Pass 기준에는 직접 포함하지 않는다.

| 기능 | 처리 기준 |
| --- | --- |
| 영상 생성 | 3주차 A등급 목표 기능으로 관리 |
| 영상 상태 조회 | 3주차 영상 생성 흐름에서 구현 |
| 역프롬프트 | S등급 도전 기능으로 분리 |
| 프롬프트 재생성 | 역프롬프트 이후 확장 기능 |
| 다운로드 | 핵심 흐름 완료 후 선택 기능 |
| 검색·필터 | 라이브러리 기본 조회 완료 후 선택 기능 |

## 3. 6월 30일 전 PM 사전 작업 기준

6월 30일 전까지의 PM 사전 작업은 최종 발표 산출물을 완성하는 단계가 아니라, **팀원이 합류했을 때 바로 개발에 들어갈 수 있도록 기준을 정리하는 단계**로 본다.

### 3.1 우선 정리할 문서

| 문서 | 목적 | 상태 |
| --- | --- | --- |
| PRD | 프로젝트 목표와 MVP 범위 정리 | 기존 문서 기반 보완 |
| MVP 기능 범위표 | 2주차 Pass, 4주차 A, S 도전 범위 구분 | 2주차 Pass 범위 우선 확정 |
| 평가 기준 매핑표 | 기능별 평가 항목 연결 | 중간점검 기준부터 정리 |
| 1주차 역할별 작업표 | 팀원별 시작 작업 범위 공유 | 담당자 확정 전 초안 상태 |
| 리스크 관리표 | API, 비용, 저장, 배포 리스크 관리 | 비용 문서 반영 필요 |

### 3.2 아직 확정하지 않아도 되는 항목

아래 항목은 프로젝트 책임자 또는 팀원과 논의 후 확정한다.

| 항목 | 이유 |
| --- | --- |
| 팀원별 세부 담당자 배정 | 실제 팀원 가능 시간과 역량 확인 필요 |
| Supabase 실제 연동 시점 | 2주차 전 mock 저장 허용 여부 논의 필요 |
| 비용 신청 및 hard limit 설정 담당 | 프로젝트 책임자와 운영 기준 확인 필요 |
| S등급 기능 착수 시점 | A등급 핵심 기능 안정화 이후 결정 |

## 4. 중간점검 시연 흐름의 의미

중간점검 시연 흐름은 6월 30일까지 반드시 제출해야 하는 공식 산출물이라기보다, **2주차 Pass 기능을 개발팀과 UX/UI가 같은 기준으로 이해하도록 돕는 내부 기준 문서**에 가깝다.

즉, 이 문서는 "발표용 대본"이라기보다 다음 질문에 답하기 위한 기준이다.

```text
2주차에 평가자 앞에서 어떤 순서로 기능이 동작해야 Pass라고 볼 수 있는가?
```

### 4.1 2주차 중간점검 시연 예시

```text
1. 사용자가 생성 화면에 들어간다.
2. 이미지 탭에서 방송 콘셉트 프롬프트를 입력한다.
3. 비율 또는 품질 옵션을 선택한다.
4. 이미지 생성 버튼을 클릭한다.
5. 로딩 상태가 표시된다.
6. 생성된 이미지가 결과 영역에 표시된다.
7. 사용자가 저장 버튼을 클릭한다.
8. 저장 완료 상태가 표시된다.
9. 라이브러리 화면으로 이동한다.
10. 방금 저장한 이미지가 카드 형태로 표시된다.
```

이 흐름은 2주차 개발 목표를 맞추기 위한 내부 기준으로만 사용하고, 6월 30일 전 공식 산출물 범위에는 포함하지 않아도 된다.

## 5. PM 관점 우선순위

현재 PM 사전 작업의 우선순위는 다음과 같이 둔다.

| 우선순위 | 작업 | 이유 |
| --- | --- | --- |
| 1 | 이미지 API 기준을 GPT Image 2.0으로 통일 | 개발 방향 혼선 방지 |
| 2 | 2주차 Pass 범위 확정 | 중간점검 실패 리스크 감소 |
| 3 | 기능별 평가 기준 매핑 | 무엇을 만들면 Pass/A/S인지 명확화 |
| 4 | 1주차 역할별 작업표 초안 | 팀원 합류 직후 실행 가능하게 준비 |
| 5 | 리스크 및 비용 기준 정리 | API 비용 초과와 일정 지연 방지 |

## 6. 팀 공유용 한 줄 정리

A2팀은 이미지 생성 API를 **GPT Image 2.0**으로 통일하고, 2주차 중간점검에서는 **프롬프트 입력 → 이미지 생성 → 결과 표시 → 저장 → 라이브러리 조회** 흐름을 Pass 기준으로 우선 완성한다.
