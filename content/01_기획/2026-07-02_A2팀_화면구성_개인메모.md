# A2팀 화면 구성 개인 메모

작성일: 2026-07-02  
용도: 2026-07-03 PM·UX/UI 화면 구성 논의 전 개인 정리  
문서 상태: 개인 초안  

---

## 1. 먼저 정해야 할 것

내일 화면 구성 논의에서는 화면을 예쁘게 그리는 것보다, 사용자가 어떤 순서로 서비스를 쓰는지 먼저 잡는 게 중요해 보인다.

핵심 흐름은 아래처럼 두면 된다.

```text
홈
→ 이미지 생성
→ 결과 확인 및 저장
→ 라이브러리 조회
→ 영상 생성
→ 영상 결과 저장
→ 라이브러리에서 다시 확인
```

초기 버전에서는 화면을 너무 많이 쪼개기보다, 사용자가 길을 잃지 않게 만드는 것이 우선이다.

---

## 2. 전체 화면 목록 초안

| 화면 | 우선순위 | 역할 | 메모 |
| --- | --- | --- | --- |
| 온보딩 / 랜딩 | 낮음 | 처음 들어온 사용자에게 서비스 소개 | MVP에서는 홈에 간단히 섞어도 될 듯 |
| 홈 | 높음 | 작업 시작 지점 | 이미지 생성, 영상 생성, 라이브러리 진입을 바로 보여주기 |
| 로그인 | 중간 | 기존 사용자 진입 | 이메일 또는 소셜 로그인 중심 |
| 회원가입 | 중간 | 신규 사용자 진입 | 최대한 짧게 구성 |
| 이미지 생성 | 매우 높음 | 프롬프트 입력 후 이미지 후보 생성 | 2주차 Pass 핵심 화면 |
| 이미지 결과 / 편집 | 매우 높음 | 생성 결과 확인, 저장, 재생성, 간단 편집 | 별도 편집기보다 생성 화면 안에 붙이는 쪽이 좋아 보임 |
| 영상 생성 | 높음 | 3x3 스토리보드 이미지와 프롬프트로 영상 생성 | 4주차 A등급 핵심 화면 |
| 라이브러리 | 매우 높음 | 생성 이미지와 영상 저장·조회 | 이미지/영상 통합 라이브러리로 가는 게 자연스러움 |
| 내 프로필 / 설정 | 낮음 | 계정, 구독, 문의, 기본 설정 | MVP에서는 최소 정보만 |
| 유료 플랜 | 낮음 | 결제/구독 안내 | 초기에는 설정 안에 넣고, 나중에 별도 페이지 검토 |
| 이미지 편집기 | 보류 | 이미지 세부 수정 전용 화면 | 처음부터 별도 화면으로 만들면 범위가 커질 수 있음 |

## 3. 추천 IA 초안

내 생각에는 처음 구조를 아래 정도로 잡으면 논의하기 편할 것 같다.

```text
/
├─ 홈
├─ 이미지 만들기
│  ├─ 프롬프트 입력
│  ├─ 생성 결과
│  ├─ 저장
│  └─ 간단 편집 또는 재생성
├─ 영상 만들기
│  ├─ 3x3 스토리보드 이미지 업로드
│  ├─ 영상 프롬프트 입력
│  ├─ 생성 상태 확인
│  └─ 영상 결과 확인
├─ 라이브러리
│  ├─ 이미지
│  ├─ 영상
│  └─ 상세 보기
└─ 내 정보
   ├─ 계정
   ├─ 구독
   └─ 문의 / 설정
```

처음부터 랜딩, 가격 페이지, 별도 편집기를 모두 분리하면 구조가 커진다.  
그래서 MVP 기준으로는 `홈`, `이미지 만들기`, `영상 만들기`, `라이브러리` 네 개를 중심으로 잡는 게 좋아 보인다.

---

## 4. 화면별 생각 정리

### 4.1 온보딩 / 랜딩

따로 있어야 할지는 아직 애매하다.

외부 사용자 유입을 생각하면 필요하지만, 지금은 과제/시연 목적이 더 크기 때문에 독립 랜딩을 크게 만들 필요는 없어 보인다. 처음에는 홈 상단에 서비스 설명을 짧게 넣고, 바로 생성 화면으로 진입하게 만드는 편이 낫다.

정해야 할 것:

- 랜딩을 별도 페이지로 둘지
- 홈 안에 간단한 소개 영역만 둘지
- 비로그인 사용자도 생성 화면을 볼 수 있게 할지

### 4.2 홈

홈은 서비스 설명보다 작업 시작 지점에 가까워야 한다.

사용자가 들어오자마자 아래 선택지를 볼 수 있으면 좋다.

- 이미지 만들기
- 영상 만들기
- 최근 작업 이어서 하기
- 라이브러리 보기

처음 사용하는 사용자에게는 빈 화면 대신 간단한 시작 버튼을 보여주고, 작업 이력이 있는 사용자에게는 최근 작업물을 먼저 보여주는 식으로 구성하면 자연스럽다.

### 4.3 로그인 / 회원가입

로그인과 회원가입은 기능을 많이 넣기보다 짧게 가는 게 좋다.

MVP에서는 아래 정도면 충분해 보인다.

- 이메일 로그인
- 소셜 로그인 가능 여부 확인
- 회원가입 시 이름, 이메일, 비밀번호 정도만 입력
- 약관/동의는 필요한 만큼만

생성형 서비스라 작업물 저장이 중요하므로, 라이브러리 저장 시점에는 로그인이 필요하다는 흐름을 잡아야 한다.

### 4.4 이미지 생성

2주차 Pass의 핵심 화면이다.

이 화면에서 사용자는 캐릭터 또는 배경 이미지를 만들고, 생성 결과를 확인한 뒤 저장할 수 있어야 한다.

구성 초안:

- 생성 목적 선택: 캐릭터 / 배경 / 키이미지
- 프롬프트 입력 영역
- 이미지 비율 또는 스타일 옵션
- 생성 버튼
- 로딩 상태
- 생성 결과 후보
- 저장 버튼
- 다시 생성 버튼

처음에는 옵션을 너무 많이 두지 않는 게 좋다. 프롬프트 입력과 결과 확인이 가장 중요하다.

### 4.5 이미지 결과 / 편집

첨부한 메모에는 `이미지 생성 및 편집창`과 `이미지 편집기?`가 따로 있는데, 처음부터 완전히 분리하지 않는 쪽이 좋아 보인다.

초기에는 이미지 생성 화면 안에서 아래 기능만 제공하면 충분하다.

- 결과 이미지 크게 보기
- 선택한 이미지 저장
- 같은 프롬프트로 다시 생성
- 프롬프트 수정 후 다시 생성
- 필요 시 간단한 편집 진입

별도 이미지 편집기는 나중에 기능이 커질 때 분리해도 된다. 지금 분리하면 UX/UI와 FE 범위가 커질 가능성이 있다.

### 4.6 영상 생성

영상 생성 화면은 이미지 생성 화면과 비슷한 구조로 가면 사용자가 이해하기 쉽다.

입력값은 아래 두 가지가 핵심이다.

- B서비스에서 받은 `9개 컷이 3x3으로 들어있는 단일 스토리보드 이미지`
- 영상 생성용 프롬프트

구성 초안:

- 3x3 스토리보드 이미지 업로드
- 영상 프롬프트 입력
- 생성 버튼
- 생성 중 상태 표시
- 완료 시 영상 미리보기
- 실패 시 다시 시도
- 저장 또는 라이브러리 이동

영상은 생성 시간이 걸릴 수 있으므로, 로딩 화면보다 `queued`, `processing`, `completed`, `failed` 같은 상태를 사용자에게 쉽게 보여주는 것이 중요하다.

### 4.7 라이브러리

라이브러리는 이미지와 영상을 따로 나누기보다 통합해서 보여주는 게 좋아 보인다.

사용자는 자신이 만든 결과물을 다시 찾고, 이어서 작업하고, 다운로드할 수 있어야 한다.

구성 초안:

- 전체 / 이미지 / 영상 필터
- 생성일 기준 정렬
- 카드형 목록
- 상세 보기
- 다시 편집하기
- 영상에 사용하기
- 다운로드

2주차에는 이미지 저장·조회가 먼저 동작해야 하고, 4주차에는 영상까지 같은 라이브러리 안에 들어오면 된다.

### 4.8 내 프로필 / 설정

초기에는 최소 기능만 있으면 된다.

구성 초안:

- 계정 정보
- 구독 상태
- 문의
- 기본 생성 옵션
- 로그아웃

구독이나 결제 정보가 아직 확정되지 않았다면, 화면 구조만 잡고 실제 상세 기능은 뒤로 미뤄도 될 것 같다.

### 4.9 유료 플랜

유료 플랜은 지금 당장 별도 페이지로 크게 만들 필요는 없어 보인다.

초기에는 내 프로필 또는 설정 안에 `구독 상태` 정도로 넣고, 실제 결제 정책이 정해지면 별도 화면으로 분리하는 게 안전하다.

정해야 할 것:

- 무료 사용량 제한이 있는지
- 이미지 생성과 영상 생성의 차감 기준을 다르게 둘지
- 결제 전환 지점을 어디에 둘지

## 5. MVP 기준으로 꼭 필요한 화면

내일 논의가 길어지면, 우선 아래 화면만 확정해도 된다.

| 구분 | 화면 | 이유 |
| --- | --- | --- |
| 1순위 | 이미지 생성 | 2주차 Pass 핵심 |
| 1순위 | 이미지 결과 / 저장 | 생성 후 저장 흐름 필요 |
| 1순위 | 라이브러리 | 저장한 결과 조회 필요 |
| 2순위 | 홈 | 서비스 진입과 작업 시작 지점 |
| 2순위 | 영상 생성 | 4주차 A등급 핵심 |
| 3순위 | 로그인 / 회원가입 | 저장 기능과 연결 |
| 3순위 | 프로필 / 설정 | 계정·구독 정보 관리 |

---

## 6. 내일 회의에서 물어볼 질문

- 랜딩을 별도 페이지로 둘지, 홈 안에 흡수할지
- 이미지 생성과 이미지 편집을 한 화면에서 처리할지
- 별도 이미지 편집기를 MVP에 넣을지, 나중으로 미룰지
- 라이브러리는 이미지/영상을 한 화면에서 보여줄지
- 영상 생성 화면의 입력 구조를 이미지 생성 화면과 얼마나 비슷하게 가져갈지
- 비로그인 사용자가 어디까지 볼 수 있게 할지
- 유료 플랜은 독립 페이지로 둘지, 설정 안에 넣을지

## 7. 현재 생각

지금 단계에서는 화면을 많이 만드는 것보다, 사용자가 아래 흐름을 막힘없이 이해하는 게 더 중요하다.

```text
무엇을 만들지 선택한다
→ 프롬프트를 입력한다
→ 결과를 확인한다
→ 마음에 드는 결과를 저장한다
→ 라이브러리에서 다시 본다
→ 필요하면 영상 생성으로 이어간다
```

그래서 내 기준 추천은 다음과 같다.

```text
홈
이미지 만들기
영상 만들기
라이브러리
내 정보
```

랜딩, 가격 페이지, 별도 이미지 편집기는 처음부터 크게 가져가기보다 MVP 이후 분리 여부를 다시 판단하는 게 좋아 보인다.
