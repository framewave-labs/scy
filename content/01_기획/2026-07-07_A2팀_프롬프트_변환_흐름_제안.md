## 1. 배경

프롬프트 가이드 작업 중, 사용자가 한국어로 입력한 요청을 GPT Image 2에 그대로 넣지 않고 "한국어 입력 → 구조화된 영어 프롬프트 변환 → 사용자 확인/수정 → 이미지 생성" 흐름으로 만드는 게 나을지 검토했다. 단순 번역이 아니라 의도·대상·스타일·구도·제약조건을 정리해주는 것이 목적이며, 특히 키이미지·캐릭터 시트·배경 시트처럼 일관성이 중요한 작업에서 효과가 크다.

## 2. IA 1차 확정본 대조 결과

`2026-07-03_생성형AI통합엔진_A2_IA.xlsx`의 IA 시트를 확인한 결과, **이 변환/미리보기 단계는 현재 확정본 어디에도 없다.**

| 확인 위치 | 현재 상태 |
| --- | --- |
| IA-017 (이미지 생성 화면 > 프롬프트 입력) | "이미지 생성용 프롬프트"로만 기술, 변환/미리보기 단계 없음. IA-017 다음이 바로 IA-018(옵션 선택)·IA-020(생성 중 상태) |
| IA-034 (영상 생성 화면 > 영상 프롬프트 입력) | 동일하게 변환 단계 없음 |
| IA-022~025 (방향별 레퍼런스, A001) | "기준 이미지 선택" 하나만 있고, 캐릭터/배경 시트 재생성 시 "고정 요소"를 구분하는 개념이 없음 |
| 후속안건 시트 | "사전 프롬프트 가이드 — 룰 설정(네거티브 키워드 등 전역 기본값)" 항목은 있으나, 이는 기본값 주입 논의이지 번역/변환 미리보기와는 다른 안건 |

즉 이 아이디어는 기존 IA의 어떤 항목과도 겹치지 않는 **완전히 새로운 제안**이다.

## 3. 범위 판단 — 2주차 Pass에는 영향 없음

`패스범위` 시트 기준 M001(이미지 생성 프롬프트 입력)의 완료 기준은 "텍스트 입력 가능"이 전부다. 즉 **2주차 Pass 기준에는 변환 단계가 필요 없다.** 이 제안은 A등급 이후 품질·일관성을 높이는 강화 요소로 제안하는 것이며, 2주차 진행을 막거나 늦추는 항목이 아니다.

## 5. 진행 방식 제안

### 5.1 깊이는 2단계로 제한
```
한국어 입력 → [Claude Sonnet: 구조화된 영어 프롬프트 변환] → 사용자 프리뷰/수정 → 승인 → GPT Image 2 호출
```
복잡한 멀티스텝 자동 편집은 MVP 제외 범위(패스범위 시트 기준)와 맞지 않으므로 2단계로 제한한다.

### 5.2 번역 모델은 Claude Sonnet — 이미 예산에 있음
[A2팀 최종 API 비용 산정](https://framewave-labs.github.io/framewave-docs/00_%ED%86%B5%ED%95%A9%EB%AC%B8%EC%84%9C/2026-07-06_a2%ED%8C%80_%EC%B5%9C%EC%A2%85_api_%EB%B9%84%EC%9A%A9_%EC%82%B0%EC%A0%95)에 "Claude Sonnet — 프롬프트 생성 보조" 350회가 이미 반영되어 있다. 이 기능은 새 예산이 아니라 **이미 계획된 항목을 실제 화면 흐름으로 formalize하는 것**이다.

### 5.3 프리뷰/승인이 필요한 이유
이 프로젝트는 "재생성도 전부 과금"을 비용 원칙으로 계속 유지해왔다. 변환 결과를 보여주지 않고 바로 이미지를 생성하면, 변환이 잘못됐을 때 상대적으로 비싼 이미지 생성 비용(장당 $0.042~0.167)까지 낭비된다. 텍스트 변환은 상대적으로 저렴하므로 여기서 한 번 확인시키는 것이 비용 관리 원칙과도 맞는다.

### 5.4 시트 작업의 "고정 요소" 개념
캐릭터 시트를 여러 장 생성할 때 인물 특징은 고정하고 포즈·구도만 바꾸는 경우가 많다. 완전한 필드별 구조화 스키마까지 가면 MVP치고 과하므로, **"고정 텍스트 블록 + 가변 텍스트 블록" 2분할** 정도로 제한하는 것을 제안한다.

## 6. 개발자(BE)에게 확인이 필요한 것

1. 번역 호출을 별도 API로 분리할지 (`POST /api/prompt-translate`) 아니면 `generate-image`에 통합할지
2. `MediaFile`/`GenerateJob`에 원문(한국어) / 변환 초안(영어) / 최종 승인 프롬프트(영어) 3가지를 어떻게 저장할지 — 필드 추가 여부
3. 캐릭터 시트 재생성 시 "고정 블록"을 어디서 관리할지 (프론트 상태 vs BE 저장)
4. 번역 실패 시 폴백 정책 (원문 그대로 노출 vs 에러 처리)
5. 동일 입력 재요청 시 번역 결과 캐싱 여부 (비용 절감)

## 7. 디자이너(UX/UI)에게 요청할 것

1. 입력 → 변환 프리뷰 → 승인 화면 흐름 와이어프레임 (IA-075~078 기준)
2. 원문/변환본 비교 UI — 변환본은 읽기 전용이 아니라 직접 수정 가능한 편집 영역으로 설계
3. "프롬프트 변환 중"과 "이미지 생성 중" 로딩 상태를 별도로 구분
4. 캐릭터/배경 시트 재생성 시 "고정 요소"를 시각적으로 구분해 보여주는 UI (예: 잠금 아이콘)
5. **(07-07 추가)** 토글 켜짐/꺼짐 상태에 따른 화면 차이 — 꺼짐일 때는 원문 그대로 생성, 켜짐일 때만 변환 프리뷰 노출
