# A2팀 사용자스토리 v0.1

> 기준 문서: 최종 IA v2.1 및 `2026-07-13_A2팀_유즈케이스_v0.1`
> 작성자: 손찬용 PM
> 목적: 기능을 사용자 가치 단위로 나누고, 개발·QA가 같은 완료 조건을 확인하도록 한다.

## 1. 작성 규칙

- 사용자스토리 형식: **[사용자]로서 [목표]하고 싶다. 그래야 [가치]를 얻을 수 있다.**
- `확정`은 최종 IA 기준 구현·검토 대상이다.
- `확인 중`, `후속안건`, `S등급 도전`은 정책 또는 우선순위가 확정되기 전까지 완료 약속으로 보지 않는다.
- 우선순위는 `P0` 핵심 완주, `P1` 주요 사용성, `P2` 확장·후속으로 구분한다.

## 4. 공통 완료 기준

| 구분 | 완료 기준 |
| --- | --- |
| 정상 흐름 | 시작 조건부터 결과 표시·다음 액션까지 끊김 없이 수행된다. |
| 입력 검증 | 오류가 있는 필드를 식별하고 사용자의 기존 입력을 유지한다. |
| 로딩·처리 | 중복 요청을 방지하고 현재 상태를 사용자에게 표시한다. |
| 실패·재시도 | 실패 이유를 안내하고 재입력 없이 재시도할 수 있다. |
| 저장 | 자동 저장 대상은 별도 저장 버튼 없이 라이브러리에서 확인된다. |
| 접근 권한 | 로그인 전용 화면은 비로그인 사용자가 직접 접근할 수 없다. |
| 추적성 | 사용자스토리 ID가 유즈케이스·기능 ID·IA ID와 연결된다. |

## 5. Definition of Ready

- IA 근거와 범위 상태가 표시돼 있다.
- UX/UI 화면 또는 컴포넌트 위치가 확인돼 있다.
- 필요한 API·데이터가 확정됐거나 `BE 확인`으로 분리돼 있다.
- 정책 미확정 항목은 담당자와 결정 질문이 적혀 있다.

## 6. Definition of Done

- 인수조건을 QA가 Pass/Fail로 판단할 수 있다.
- 정상·로딩·빈 상태·실패·재시도 흐름이 구현됐다.
- 실제 API 연결 또는 mock 상태가 명시돼 있다.
- 관련 IA·기능명세서·테스트 항목이 같은 결정으로 갱신됐다.
