# A2팀 키이미지 정의 (PD 확인)
> 작성일: 2026-07-07 | 작성자: 손찬용 PM
> 상태: **PD 확정** — 이슬기 PM이 PD에게 직접 확인받은 내용을 기준으로 정리
> 관련 문서: `2026-07-07_A2팀_프롬프트_변환_흐름_제안.md`, `A2팀_2026-07-07_회의록.md`

## 1. 배경 — 무엇을 확인했나

이슬기 PM이 PD에게 아래 두 가지를 직접 문의했다.
1. 키이미지가 캐릭터/배경 **각각** 만들어지는 것인지, 아니면 캐릭터+배경이 **하나로 통합**되어야 키이미지인지
2. A 기능 단계에서 다양한 앵글(다양한 포즈/다양한 각도의 배경)로 출력되길 원하는 게 맞는지

같은 시점에 B2팀(스토리보드 빌더 담당 PM 신채연·오현석)도 "A라인에서 B로 넘어오는 키 이미지가 ① 인물+배경이 합쳐진 완성 사진 1장인지, ② 인물/배경이 각각 따로인지"를 문의해온 상태였다 — 우리만의 혼동이 아니라 크로스팀 공통 확인 사항이었다.

## 2. PD 답변 (확정)

### 2.1 키이미지 = 캐릭터 1명에 대한 고정 기준 이미지
- 키이미지는 **캐릭터+배경이 합쳐진 장면 이미지가 아니다.**
- 하나의 이미지 안에 아래 요소가 전부 포함된, **캐릭터 전용 고정 레퍼런스 시트**다.
  - 전방향 풀샷 (정면/측면/후면 턴어라운드)
  - 표정 배리에이션
  - 액세서리·소품 디테일
  - 스타일, 키컬러(컬러 팔레트)
- 이 이미지를 이후 생성 요청에 계속 참조로 넣어서 **일관성을 유지**한다. 첫 캐릭터 이미지를 생성하면 보통 이 키이미지 형태 그대로 나온다.
- 세부 디테일(포즈, 액션, 장면 맥락)은 키이미지에 넣지 않고 **프롬프트 텍스트로 직접 지시**한다.
- 참고 예시 3장을 PD가 직접 제공: 판타지풍 캐릭터 디자인 시트(Vessa Morivan), 애니메이션풍 캐릭터 설정표(미라 한), 실사 인물 4앵글 턴어라운드 — 형식은 다르지만 공통적으로 "턴어라운드 + 표정/디테일 + 컬러 팔레트"를 한 장에 담는 구조.

### 2.2 배경은 360도(다각도) 불필요
- 배경은 캐릭터처럼 전방향 시트가 필요 없다.
- **특징적인 요소만 잡아주면 나머지는 자연스럽게 생성된다** — 즉 배경 시트는 캐릭터 시트보다 훨씬 가볍게 가져가도 된다.

### 2.3 예시 프롬프트 템플릿 (PD 제공, 원문 그대로)

PD가 위 캐릭터 키이미지(미라 한 예시)를 만들 때 쓴 프롬프트 구조를 그대로 공유해줬다. 2단 구성이다.

**Part 1 — Prose Description**: 이미지 전체를 자연어로 설명하는 문단 (피사체, 레이아웃, 톤앤매너 요약)

**Part 2 — JSON 구조 필드**

| 필드 | 내용 |
| --- | --- |
| SUBJECT | 인물 기본 정보, 헤어/외형, 표정 배리에이션, 포즈(턴어라운드 각도) |
| CLOTHING & STYLING | 의상 아이템별 소재·색상(헥스코드 포함) |
| ACCESSORIES | 액세서리별 소재·색상 |
| SCENE | 장면/레이아웃 유형, 핵심 요소 |
| BACKGROUND | 배경 요소 (그리드, 텍스트 블록, 컬러 스와치 등) |
| LIGHTING | 조명 방식 |
| TECHNICAL | 촬영/렌더링 기술 정보 (해당 없으면 N/A) |
| COMPOSITION | 구도 방식 (그리드형, 정면/직교 투영 등) |
| STYLE | 아트 스타일, 주요 색상 |
| ASPECT RATIO | 출력 비율 |
| CONSTRAINTS - Must keep | 반드시 유지해야 할 요소 (헤어스타일, 상징 요소, 의상 특징 등) |
| CONSTRAINTS - Avoid | 피해야 할 것 (블러, 비현실적 피부톤, 왜곡된 해부구조 등) |

**원문 예시 (미라 한 캐릭터 키이미지)**:

```

BACKGROUND: grid lines, text blocks, color palette swatches

LIGHTING: studio lighting, even, flat, diffused
```

## 3. 기존 이해와 달라진 점

| 항목 | 기존 이해 (07-06까지) | PD 확인 (07-07) |
| --- | --- | --- |
| 키이미지 정의 | 캐릭터 시트 + 배경 시트를 참고해 만드는 **통합 장면 이미지** | 캐릭터 1명에 대한 **고정 기준 이미지** (턴어라운드+표정+액세서리+컬러). 캐릭터 시트와 사실상 동일 개념 |
| 배경 시트 | 캐릭터 시트와 동일하게 다각도(360도 개념) 필요 | **불필요** — 특징 요소만 있으면 나머지는 자연 생성 |
| 캐릭터시트 → 키이미지 관계 | 시트를 만든 뒤 별도로 키이미지를 한 번 더 생성하는 2단계 | **사실상 같은 산출물** — 별도 생성 단계 아님 |
| A라인 → B팀 전달 형태 | 미정 (B2팀도 동일하게 미정 상태로 문의 중이었음) | 이번 답변으로 "인물 중심 고정 레퍼런스"라는 방향이 유력해짐. **다만 B팀 전달 형식 자체는 이 답변으로 자동 확정된 게 아니므로 B2팀에는 별도로 답변 필요** |

## 5. 아직 남은 것

PD 답변으로 "우리 서비스 안에서 키이미지가 어떤 형태로 나와야 하는가"는 정리됐지만, **B팀에 실제로 무엇을(키이미지만인지, 배경 정보도 같이인지) 어떤 형식으로 넘길지는 별도로 확정해야 한다.** B2팀 문의에 대한 회신은 이 부분까지 정리한 뒤 진행하는 것을 권장한다.
